Overview

Atelier is an autonomous, governed, multi-surface design-system generation engine. Unlike typical code-generation tools that output a single page on a best-effort basis, Atelier establishes a rigorous structural convergence loop. Given a brief, the system processes it through a multi-specialist pipeline, verifies structural and token constraints, executes accessibility evaluation, and scores survivors using a multi-judge consensus panel.

The system operates on a zero-tolerance token-fidelity model: design decisions are captured as design tokens conforming to the Design Tokens Community Group (DTCG) specification. Modifying a single token propagates change across all generated pages, ensuring complete visual coordination across the surface envelope.

security

System Integrity Objective

Atelier ensures all model allocations, token budgets, and output gates are enforced strictly. Off-token modifications or invalid semantic markup trigger fail-closed halts, ensuring that only production-grade code reaches user-facing files.

8-Node DAG & Specialist Pipeline

The execution runtime is structured as an 8-Node Directed Acyclic Graph (DAG) designed to split high-level objectives into distinct, specialist contexts:

Node Name Responsibilities
N1 Brief Parser Parses the natural language intake and translates goals into structured system constraints.
N14 Web-Research Intake (WRAI) Augments the brief using targeted queries to resolve brand assets and design standards.
N2 Source Resolver Assembles necessary code patterns, library dependencies, and asset layouts.
N3a Specialist Pipeline Orchestrates 6 design and engineering specialist agents sequentially via the ADK.
N3b Fixer Convergence Loop Executes iterative fixes targeting explicit errors raised by gates or judges.
N3c Deterministic Gates Enforces structural and stylistic conformance pre-flight before judging.
N3d Consensus Judge Gathers scores across the 5 D-O-R-A-V axes from independent evaluators.
N4 Best Pick Selector Selects the optimal candidate scoring above the convergence threshold.

The 6 DDLC Specialist Roles

Inside node N3a, a sequential pipeline uses Google ADK orchestration to route tasks through 6 specialist agents:

  1. UX Researcher: Establishes domain research, user flow goals, and specific UI requirements.
  2. IA/User-Flow Specialist: Defines semantic structure, page layout trees, and element hierarchies.
  3. Wireframer: Converts user flows into low-fidelity HTML representations.
  4. UI Designer: Establishes spacing, visual structures, typography weight, and colors.
  5. Interaction Designer: Generates responsive interactions and state transition styling.
  6. Token Generator: Compiles design outputs into strict DTCG-compliant tokens.
verifiedQuality Control

Deterministic Quality Gates

To eliminate empty skeletons, broken styling, and visual regressions, every generated candidate must clear a battery of 6 deterministic quality gates before it is eligible for evaluation by the LLM-based judge panel. If a candidate fails any gate, it is rejected and sent back to the Fixer loop:

  • Semantic HTML Gate: Verifies document structure, ensuring appropriate tag layout (e.g. <header>, <main>, <nav>) and basic structural elements.
  • CSS Validity Gate: Runs style declarations through validation engines, halting on syntax errors or invalid declarations.
  • Token Fidelity Gate: Audits custom styling to ensure colors, margins, and typography map exactly to declared design tokens.
  • Lighthouse Heuristics Gate: Audits basic document standards, performance tags, and meta tag completeness.
  • Axe Accessibility Gate: Scans the DOM tree using automated accessibility checks, rejecting any candidate with critical WCAG failures.
  • Visual Difference Gate: Compares visual changes between iterations to prevent regression.

Multi-Judge Consensus (D-O-R-A-V)

Candidates that pass all deterministic gates are evaluated by a panel of independent LLM judges. The judges score each candidate on a 5-axis scale to determine a composite score:

  • Design: Visual appeal, balance, formatting hierarchy, and modern aesthetics.
  • Originality: Avoidance of generic templates and adherence to specific brand characteristics.
  • Relevance: Conformance to user brief requirements and constraints.
  • Accessibility: High contrast ratios, accessible target tap zones, and screen-reader support.
  • Visual Clarity: Clear scannability and logical information grouping.

The composite score is compiled using a Bayesian consensus formula that weights judge reliability and per-axis metrics. A candidate is declared to have converged once the composite score reaches: 

composite_score ≥ 0.70

If no candidate meets the threshold, the Fixer loop modifies the highest-scoring candidate, targeting the specific axis that recorded the lowest score, repeating up to a hard iteration ceiling.

The DPO Flywheel

The system continuously improves its performance using a closed-loop data flywheel backed by Direct Preference Optimization (DPO). The flywheel processes data in three core phases:

  1. Observe: Trajectory telemetry (scores, votes, configurations, outputs) is recorded in real-time to BigQuery.
  2. Simulate: Preference pairs are extracted where one candidate was accepted and another rejected for the same design brief, requiring a minimum score margin.
  3. Verify: Extracted preference pairs are compiled into datasets to run Vertex AI fine-tuning, promoting successful models only if they outperform the baseline adapter.

Token Governance & Security

Atelier implements strict pre-flight token-use caps to guarantee cost control and protect backend endpoints:

  • Token-Only Cap: The system governs resource consumption using tokens, not dollars. The hard limit is set to a cumulative 5,000,000 tokens per user.
  • Fail-Closed Pre-Flight Check: Before any Vertex AI endpoint call is initiated, the governor queries the Firestore database. If the token count is exhausted or the database is unavailable, the request fails closed.
  • Authentication & Ingress: Firebase Authentication validates end-user identities, while Google Cloud Identity-Aware Proxy (IAP) blocks public access to the Cloud Run API, restricting traffic to authorized identities.
  • PII Scrubbing: OpenTelemetry span processors inspect and redact sensitive parameters before traces leave the memory space.

Architecture Decision Records

Atelier's design evolution is tracked via Architecture Decision Records (ADR). The current active logs include:

ADR ID Title Consequences & Context
ADR-0024 Governed A2UI Pins the system to A2UI v0.9 (server-to-client subset) as a flag-gated experimental feature. The main design deliverable remains portable HTML/CSS to avoid coupling.
ADR-0025 Generation Latency Optimizes node concurrency in the specialist pipeline, introducing async step batches to lower pipeline execution times.
ADR-0026 OpenAPI Prod Gating Disables public access to the interactive OpenAPI UI in production, enforcing pre-auth gateways for all API routes.
